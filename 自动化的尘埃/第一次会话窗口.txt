## 可行性判断：完全可行

你这个需求的本质是：**让电脑在没人用的时候自动帮你清原神日常**。技术上有两条成熟路径，实现成本都不高。

### 路径 A：Windows 计划任务「空闲时触发」（最省事）

Windows 任务计划程序原生支持「On idle」触发器。你只需配置：
- 触发器：计算机空闲超过 X 分钟（如 10 分钟）
- 条件：仅在空闲后启动，用户活动时停止
- 每日重置：结合一个标记文件，确保一天只跑一次

**优点**：零代码，纯系统能力。**缺点**：Windows 的"空闲"定义比较粗糙（只检测键鼠输入），而且任务计划程序有一个老旧限制——它要求计算机在空闲检测期间持续空闲，实际触发可能需要长达 15-30 分钟。

### 路径 B：编写一个轻量级空闲监控程序（更灵活、推荐）

写一个 PowerShell 脚本（约 60-80 行），后台运行，轮询检测：
- 用户空闲时间（调用 Windows API `GetLastInputInfo`，精确到秒）
- 今日是否已执行（写入注册表或本地文件做标记）
- 可选：检测原神/BetterGI 是否已在运行（避免重复启动）
- 满足条件时以管理员身份拉起你的 bat 脚本

这个方案可以用任务计划程序在开机时启动，一直在后台低负载运行，检测更精准，逻辑也完全可控。

**优点**：精确控制空闲时长阈值、每日限制、冲突检测。**缺点**：需要写一个小程序。

## 需要注意的风险点

1. **管理员权限**：你的 bat 脚本需要管理员权限才能静音/恢复音频服务。无论选哪条路径，拉起脚本时都必须提权，会弹出 UAC 提示。如果你的电脑日常以管理员账户运行且 UAC 设为"不通知"，则无此问题。
2. **BetterGI 需要游戏已在运行**：脚本只是启动 BetterGI 的自动化流程，它不会帮你打开原神。你需要确保游戏已经在运行（可能是开机自启了启动器+游戏）。
3. **空闲检测的边界**：如果电脑在播放视频、下载文件，Windows 认为键鼠无操作就是"空闲"，但实际上你可能不希望 BetterGI 在下载时抢占资源。路径 B 可以加入 CPU/GPU 占用检测来规避。
4. **联想工具箱**：你提到的联想工具箱可能有系统性能模式切换功能。如果它在空闲时切到省电模式，可能会影响原神和 BetterGI 的运行。需要在触发前确认电源模式。

## 修订方案

**核心逻辑变化**：标记文件不再只记日期，而是记录**上次执行的具体时间戳**（精确到秒）。每次启动时计算距离上次执行过去了多久，不足 20 小时则直接退出，不进入空闲监控。

### 参数配置

- 空闲检测阈值：15 分钟
- 轮询间隔：1 分钟
- 最小执行间隔：20 小时（72000 秒）
- 标记文件：`D:\MIHOYO\GenShinTool\.bettergi_last_run`（内容为 ISO 8601 时间戳，如 `2026-06-06T22:30:00`）
- 启动目标：桌面快捷方式 `C:\Users\32683\Desktop\RunBetterGI.bat - 快捷方式.lnk`

### 执行流程

```
开机登录 → 计划任务启动 IdleMonitor.ps1
              │
              ├─① 读取标记文件中的上次执行时间
              │    └─ 距上次不到 20 小时 → 直接退出，不监控
              │
              ├─② 检查 BetterGI.exe 是否已在运行
              │    └─ 已在运行（比如你手动启动了）→ 直接退出
              │
              └─③ 进入空闲监控循环（每 60 秒检测一次）
                   │
                   └─ 空闲 ≥ 15 分钟 →
                        写入当前时间戳到标记文件
                        启动桌面快捷方式
                        脚本自身退出（释放所有内存）
```

### 设计要点

**为什么用时间戳而不是日期**：假设你周一晚上 10 点执行了一次，周二早上 8 点开机。日期标记判断"今天是周二，昨天是周一，不一样，可以跑"——但实际才过了 10 小时，树脂还没回满。时间戳方案会算出只过了 10 小时 < 20 小时，拒绝执行。等到周二晚上 6 点以后，距离上次超过 20 小时，才会进入空闲监控。

**BetterGI 进程检测**：如果你手动双击快捷方式启动了 BetterGI，空闲监控脚本检测到进程已在运行就会直接退出。这样避免了你手动跑完之后，电脑进入闲置状态又自动拉起一份。

**零残留**：触发后脚本立即退出。如果电脑一整天都没空闲到 15 分钟，脚本虽然会一直运行但极轻量（每 60 秒才醒来一次检查，其余时间系统挂起，内存占用约 20-30 MB）。

**一天多次开机**：每次开机都会先检查时间戳，不足 20 小时直接退出，不会重复执行。

## 问题所在

## 修正后的流程

脚本**在任何情况下都不提前退出**（除非检测到 BetterGI 已经在运行）。它始终驻留在后台循环中，每 60 秒同时检查两个条件：

```
开机 → IdleMonitor.ps1 启动
         │
         ├─ 读取标记文件中上次执行时间
         │
         └─ 进入后台循环（每 60 秒检查一次）：
              │
              ├─① BetterGI.exe 已在运行？
              │    └─ 是 → 你手动启动了，脚本退出
              │
              ├─② 距上次执行 ≥ 20 小时？
              │    └─ 否 → 回到循环顶部，继续等
              │
              ├─③ 系统空闲 ≥ 15 分钟？
              │    └─ 否 → 回到循环顶部，继续等
              │
              └─ 条件②和③都满足 →
                   写入当前时间戳到标记文件
                   启动桌面快捷方式
                   脚本自身退出（释放内存）
```

**两个条件必须同时成立才触发**，缺一个就继续轮询。不存在"等待唤醒"的问题——脚本本身就是那个一直在等的角色。

- 周二 08:00 开机，脚本启动，读标记：距上次 10h < 20h → 进入循环
- 周二 08:00 ~ 18:00，每 60 秒检查一次，gap 始终 < 20h → 不触发但**始终在循环内**
- 周二 18:00，gap 达到 20h，但你在用电脑（空闲 < 15min）→ 不触发
- 周二 19:00 你离开，空闲达到 15min → gap 21h ≥ 20h → 触发

> 关键点：18:00 满 20 小时的那一刻，脚本**正在循环中**，下一轮（最多 60 秒后）就会检测到条件变化。不存在"没人唤醒"的问题。

**场景 B：一天内多次开机**

- 周一 22:00 触发，标记写入
- 周二 08:00 开机，gap 10h → 循环中
- 周二 09:00 因故重启，开机后脚本再次启动
- 读标记：距离周一 22:00 仍是 11h < 20h → 循环中
- 周二 19:00 触发

> 标记文件的时间戳是**绝对时间**，跨重启不丢失，计算始终正确。

**场景 C：你手动双击了快捷方式**

- 脚本在后台循环（gap 15h < 20h）
- 你手动双击桌面快捷方式，BetterGI 启动
- 下一轮循环（60 秒内）：`Get-Process BetterGI` 发现进程存在
- 脚本判定为手动操作 → 写入当前时间戳到标记 → 退出
- 标记时间变为"现在"，20 小时倒计时重新开始

> 不会出现"你手动跑完、脚本又自动跑一次"。

**场景 D：标记文件损坏或丢失**

- 文件不存在或内容无法解析 → 视为"从未执行过"，gap 按无穷大处理
- 进入空闲监控，满足 15 分钟空闲即触发
- 最坏情况：比预期早触发一次（反正也是首次），后续 20 小时间隔恢复正常

**场景 E：电脑休眠 / 合盖**

- 系统休眠时 `Environment.TickCount` 和 `GetLastInputInfo` **同步暂停**，不会错误累计"空闲时间"
- 唤醒后：若你动了键鼠，idle 归零重新计算；若没动，idle 维持休眠前的值，继续累计
- 行为符合直觉

**场景 F：电脑全天不空闲**

- 脚本从开机待到关机，每 60 秒检查一次，从未触发
- 关机时随系统终止
- 第二天开机重新来

## 结论

**没有遗留的"无法唤醒"问题。** 脚本只有三种退出路径：

1. 条件满足 → 触发执行后退出（任务完成）
2. 检测到 BetterGI 已在运行 → 判定手动操作后退出（防重复）
3. 系统关机 → 随系统终止（正常结束）

除此之外，**脚本永不退出**。它用每 60 秒一次的轮询充当唤醒机制，任何时候条件满足都会在下一轮被捕获。
